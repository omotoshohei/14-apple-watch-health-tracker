## アーキテクチャパターン

本システムは、ユーザーのローカルマシンで完結するバッチスクリプト形式のツールです。要件に基づき、開発と配布の容易さを重視して**単一のPythonスクリプトファイル (`src/health_monthly_report.py`)**として実装します。

ただし、保守性とテスト性を考慮し、スクリプトの内部構造は以下の論理的なレイヤー（責務）に分離した関数およびクラスで構築します。

```
┌──────────────────────────────────────────────────────────┐
│  src/health_monthly_report.py (エントリーポイント & 制御) │
└──────────────────────────────────────────────────────────┘
                            │ (処理フロー制御)
                            ▼
┌──────────────────────────────────────────────────────────┐
│  1. Parser / Data Loader (XMLストリーミングパース)       │
├──────────────────────────────────────────────────────────┤
│  2. Aggregator (pandasによるデータフィルタ・集計・欠損処理)│
├──────────────────────────────────────────────────────────┤
│  3. Graph Generator (Matplotlibによるグラフ描画・画像保存)  │
├──────────────────────────────────────────────────────────┤
│  4. Insight Generator (google-genai による Gemini連携)    │
├──────────────────────────────────────────────────────────┤
│  5. HTML Generator (Jinja2 による HTMLレンダリング)       │
└──────────────────────────────────────────────────────────┘
```

### レイヤー別の設計と責務

#### 1. Parser / Data Loader
- **責務**: `export.xml` から必要なヘルスケアデータを抽出し、メモリ効率の良いストリーム形式で読み込む。
- **実装方針**: `xml.etree.ElementTree.iterparse` を使用して、必要な Record 要素のみをオンメモリに展開し、使い終わった要素は即座にクリアすることで、メモリ消費を最小限に抑える。

#### 2. Aggregator
- **責務**: 抽出したデータを pandas DataFrame に変換し、指定された対象年月のデータにフィルタリングした上で、指標ごとに日別集計を行う。また、未装着日を「欠損値」として正しくマークする。
- **実装方針**: 指標別の集計ルール（単純合計か、時間ユニーク数か等）に基づいて集計し、対象月の日付マスタを基準に左結合（Left Join）することで欠損日を検出する。

#### 3. Graph Generator
- **責務**: pandas DataFrame から各指標の日別棒グラフを描画し、目標ラインを重ね合わせた上で、PNG画像として保存する。
- **実装方針**: Matplotlib を使用し、ダークテーマ（背景色 `#0f172a` 等）に合わせたカラースキームで描画する。

#### 4. Insight Generator
- **責務**: 指標ごとの統計データおよび日別の数値を基に、Gemini API を呼び出して英語のインサイトテキストを生成する。
- **実装方針**: 統計情報を含んだプロンプトを構築し、エラー発生時はプレースホルダーテキストを返すことで、処理全体が停止するのを防ぐ。

#### 5. HTML Generator
- **責務**: 生成されたグラフ画像へのパスと、Geminiから返されたインサイトテキストを Jinja2 テンプレートに流し込み、最終的なHTMLファイルを出力する。

## データ永続化戦略

### ストレージ方式
本システムはデータベースを使用せず、ローカルのファイルシステムをデータ永続化および成果物保存のストレージとして利用します。

| データ種別 | 配置先 | フォーマット | 理由 |
|-----------|----------|-------------|------|
| 元データ | プロジェクトルート (`export.xml`) | XML | AppleヘルスケアAppからエクスポートされる標準形式。 |
| グラフ画像 | `output/assets/[metric].png` | PNG | HTMLに埋め込むための軽量な画像フォーマット。 |
| 成果物 | `output/apple_watch_health_monthly_report_YYYY_MM.html` | HTML | ユーザーがブラウザで直接閲覧できるスライド形式。 |

### バックアップ・データ管理
- ローカルツールであるため、自動バックアップ機能は実装しません。
- 出力ファイル名に年月 (`YYYY_MM`) を含めることで、過去のレポートが上書きされるのを防ぎます。
- 個人情報を含む `export.xml` や、生成された `output/` ディレクトリは、Git リポジトリにコミットされないよう `.gitignore` で除外します。

## パフォーマンス要件

### レンスポンスタイムとメモリ消費

| 項目 | 目標基準 | 測定環境 / 条件 |
|------|---------|---------|
| ローカル処理時間 | 30秒以内 | 通常のMac環境、`export.xml` が 500MB 程度。Gemini API の応答時間は外部依存のため別計測。 |
| XMLパース時の追加メモリ使用量 | 150MB 以下を目安 | `xml.etree.ElementTree.iterparse` によるストリーミングパース時。pandas集計後の総メモリ使用量は抽出対象レコード数に依存する。 |

## セキュリティアーキテクチャ

### 1. APIキー管理
- Gemini APIキーは、環境変数 `GEMINI_API_KEY` から読み込みます。
- コード内にAPIキーを直接記述（ハードコーディング）することは禁止します。
- スクリプト起動時に `GEMINI_API_KEY` の存在チェックを行い、未設定の場合はわかりやすいエラーメッセージを出力して終了します。

### 2. 個人データの保護
- Apple Watchのデータには、ユーザーの健康状態や生活リズムに関する個人情報が含まれます。
- 本システムは完全にローカル環境で動作し、Gemini API へのデータ送信を除き、外部サーバーへのデータ転送は一切行いません。
- Gemini API へのデータ送信は、指標別の統計値（平均や達成率）および日別の数値データのみに限定し、ユーザー名などの直接的な識別情報は送信しません。

## スケーラビリティ設計

### 機能拡張性（指標の追加）
将来的に新しい健康指標（例: 安静時心拍数、上った階数など）をサポートする場合に、コードの変更箇所を最小限にするため、指標の定義情報を設定用のディクショナリ（メタデータ定義）として一元管理します。

```python
METRIC_DEFINITIONS = {
    'sleep_duration': {
        'name': 'Sleep Duration',
        'record_type': 'HKCategoryValueSleepAnalysis',
        'unit': 'hours',
        'target_value': 7.0,
        # ...集計ロジックやグラフ設定など
    },
    'steps': {
        'name': 'Steps',
        'record_type': 'HKQuantityTypeIdentifierStepCount',
        'unit': 'steps',
        'target_value': 8000.0,
        # ...
    }
}
```
このように設計することで、新規指標の追加時はメタデータ定義を追加し、必要に応じて集計ヘルパー関数を調整するだけで拡張可能とします。
