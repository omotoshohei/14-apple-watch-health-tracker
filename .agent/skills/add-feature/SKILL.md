# 新機能の追加スキル (Antigravity版)

このスキルは、新規機能を既存の実装パターンや設計思想に沿って、ユーザーの介入を最小限に抑えながら無停止で完全自動実装するためのガイドです。

## 概要

このスキルでは、以下の手順で新機能の実装を進めます。各プロセスにおいて、[steering](../steering/SKILL.md) スキルや [development-guidelines](../development-guidelines/SKILL.md) スキルと連携します。

---

## 実行手順

### ステップ 1: 準備とコンテキスト設定
1. 現在のタスクコンテキストを明確にします：
   - 機能名: `[ユーザーから指定された機能名]`
   - 日付: `[現在の日付をYYYYMMDD形式で取得]`
   - ステアリングディレクトリパス: `.steering/[日付]-[機能名のslug]/`
2. 上記ステアリングディレクトリを作成します。
3. 以下の3つの空ファイルを作成します：
   - `.steering/[日付]-[機能名のslug]/requirements.md`
   - `.steering/[日付]-[機能名のslug]/design.md`
   - `.steering/[日付]-[機能名のslug]/tasklist.md`

### ステップ 2: プロジェクト理解
1. プロジェクトルートの `CLAUDE.md`（存在する場合）を読み込み、プロジェクトの全体像を把握します。
2. `docs/` ディレクトリ内の永続ドキュメントを確認し、関連する設計思想やアーキテクチャを理解します。

### ステップ 3: 既存パターンの調査
1. Antigravity の検索機能を使用し、追加する機能に関連するキーワードでソースコード（`src/`）を検索します。
2. 検索結果から、既存の実装パターン、命名規則、コンポーネントの設計と利用方法を特定し、新しいコードが既存パターンと整合するようにします。

### ステップ 4: 計画フェーズ (ステアリングファイルの生成)
1. [steering](../steering/SKILL.md) スキルの **モード1: ステアリングファイル作成** の手順に従い、ステップ1で作成した3つのファイル (`requirements.md`, `design.md`, `tasklist.md`) の内容を生成・記述します。
2. **注意**: このステップが正常に完了したら、作業を中断せず、ただちにステップ5に進んでください。

### ステップ 5: 実装ループ (tasklist.md の消化)
`tasklist.md` 内の未完了タスク（`[ ]`）がすべて完了（`[x]`）するまで自動で繰り返すループ処理です。

1. **タスクリストの読み込み**:
   - `.steering/[日付]-[機能名のslug]/tasklist.md` ファイルを読み込みます。
2. **進捗の確認**:
   - 未完了タスク (`[ ]`) が存在するか確認します。
   - **未完了タスクが存在しない場合**: 実装ループを完了し、ただちに **ステップ6** へ進みます。
3. **タスクの実行**:
   - `tasklist.md` の**先頭にある未完了タスク**を特定します。
   - [steering](../steering/SKILL.md) スキルの **モード2: 実装** の原則（着手時の `（着手中）` 追記、完了時の `[x]` 更新）に従って実装を進めます。
   - コード記述の際は、[development-guidelines](../development-guidelines/SKILL.md) のコーディング規約を常に遵守します。
4. **ループの継続**:
   - タスク完了の記録後、再び `tasklist.md` の読み込み（手順1）に戻ってループを継続します。

---

### ※ 実装ループ内の例外処理ルール ※
実装ループの実行中に問題が発生した場合は、自律的に以下のルールに従って対処し、ループを継続してください。

- **ルールA: タスクが大きすぎる場合**
  - **対処法**: 現在のタスクをより小さな複数のサブタスクに分割します。`tasklist.md` を編集して元のタスクを削除し、そこに新しいサブタスク（`[ ]`付き）を挿入します。その後、ループを継続します。
- **ルールB: 技術的理由でタスクが不要になった場合**
  - **条件**: 設計・方針変更等、明確な技術的理由がある場合にのみ適用します。
  - **対処法**: `tasklist.md` 内の該当タスクを `- [x] ~~タスク名~~（実装方針変更により不要: [理由]）` の形式で更新し、ループを継続します。
- **🚨 絶対禁止事項**:
  - 未完了タスクを理由なくスキップしたり、放置したままループを終了させること。
  - ユーザーに細かな実装上の判断を繰り返し仰ぐこと（自律的に解決可能な技術的課題は自身で判断し進める）。

---

### ステップ 6: 実装検証 (品質セルフチェック)
1. `tasklist.md` の全タスクが完了状態になっていることを最終確認します。
2. 今回実装した機能に関連するすべての変更点について、品質のセルフチェックを行います。
   - **確認項目**: コーディング規約への適合、適切なエラーハンドリング、テスト可能性、既存コードパターンとの整合性。

### ステップ 7: 自動テスト・静的解析の実行
1. プロジェクトで利用可能なテスト・静的解析コマンドを順番に実行し、すべて正常にパスすることを確認します。Python プロジェクトでは、以下を標準候補とします。
   ```bash
   pytest src/
   ruff check src/
   ruff format --check src/
   ```
2. エラーが発生した場合は問題を分析し、修正コードを適用してから、再度このステップを実行してパスさせます。

### ステップ 8: 振り返りとドキュメント更新
1. [steering](../steering/SKILL.md) スキルの **モード3: 振り返り** に従い、`tasklist.md` の「実装後の振り返り」セクションを更新します（完了日、計画と実績の差分、学び、次回への改善提案）。
2. 今回の変更がプロジェクトの基本設計やアーキテクチャに影響を与える場合は、`docs/` 配下の関連永続ドキュメント（`docs/architecture.md` など）を更新します。

---

## 完了条件

このワークフローは、以下の条件をすべて満たした時点で完了となります。
- [ ] `tasklist.md` のすべてのタスクが完了状態（`[x]` または正当な技術的理由によるスキップ）になっていること。
- [ ] 自動テスト・静的解析（`pytest`, `ruff check`, `ruff format --check`）がエラーなしで成功すること。
- [ ] `tasklist.md` の振り返りセクションが記入されていること。
